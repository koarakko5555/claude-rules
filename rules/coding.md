# コーディング規約

## 全般

- コメントは原則書かない。書くのは WHY が自明でない場合のみ（隠れた制約、特定バグの回避策、意外な不変条件）。
- 複数行のドキュストリング・コメントブロックは書かない。
- 後方互換シム、未使用 `_var` リネーム、`// removed` のような墓石コメントは残さない。
- 似たコード3行は、早すぎる抽象化よりマシ。
- 起こり得ないケースのエラーハンドリングは書かない。フレームワークの保証は信頼する。
- バリデーションはシステム境界（ユーザー入力・外部API）でのみ行う。

## 変更方針

- 新規ファイル作成より既存ファイルの編集を優先する。
- バグ修正のついでに周辺をリファクタしない。タスクのスコープを守る。
- 機能追加・抽象化・リファクタを、要求された以上に行わない。
- 後方互換のための feature flag やシムを足すくらいなら、コードを直接書き換える。

## セキュリティ

- SQL injection / XSS / command injection / SSRF を絶対に作り込まない。
- シークレットをハードコードしない。環境変数を使う。
- ユーザー入力は shell / SQL / HTML に渡す前に必ずサニタイズする。
