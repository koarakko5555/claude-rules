バグを **根本原因 (root cause) まで遡って** 直す。表面的なつじつま合わせを禁じる。

---

## 大原則: 対症療法の禁止

以下は **修正案として提案しない / 採用しない**:

- ❌ `try/except` でエラーを握りつぶす（ログだけ吐いて続行する含む）
- ❌ 失敗したら成功するまでリトライする（race condition の隠蔽）
- ❌ タイムアウトを伸ばして race を覆い隠す
- ❌ 個別ケースだけ if 分岐で回避する（同じ原因の別ケースが残る）
- ❌ null/undefined が来た理由を調べずに `?? defaultValue` や `if (x) { ... }` を足す
- ❌ バリデーションを緩めて通す
- ❌ テストを書き換えて通す（テストが正しいのに実装が壊れているとき）
- ❌ 「とりあえず動くようになった」で終わる

これらが**唯一の現実的な対策**だと判断した場合のみ、**ユーザーに対症療法であることを明示**して採否を仰ぐ。暗黙に採用しない。

---

## フェーズ1: 再現

まずユーザーから以下を引き出す（既に判明していれば飛ばす）:

- **症状**: 具体的に何が起きるか（エラーメッセージ・スタックトレース・期待動作との差分）
- **再現手順**: どう操作すれば必ず起きるか
- **環境**: ローカル / staging / production、最近変えたもの
- **発生頻度**: 100% / 散発的 / 特定条件のみ

**再現できない or 情報不足のまま先に進まない**。ログ・スクショ・最小再現コードを求める。再現が散発的なら「いつ起きて、いつ起きないか」の差分を聞く。

その上で、現環境で再現を試みる:

```bash
# 例: 失敗するコマンド/テストを実行
```

再現できたら次へ。できなければ「環境差・データ差・タイミング差」のどれかが原因なので、その軸で深掘りする。

---

## フェーズ2: 証拠収集

調査は **Explore サブエージェント**に投げてメインコンテキストを汚さない:

- 「症状に関係するファイル群と、エラーが起きるコードパス」
- 「同じ症状を扱っている既存のテスト・関連ログ出力」
- 「直近で当該コードを変更したコミット (`git log -p -- <path>`)」

並行して以下を確認:

```bash
git log --oneline -20 -- <suspect_file>     # 最近の変更
git log -S "<エラー文言>" --oneline          # その文言が初登場したコミット
git blame <file> <line>                      # 該当行の出自
```

**Bisect が有効な状況**（「以前は動いていた / いつから壊れたか不明」）なら `git bisect` を提案する。

---

## フェーズ3: 仮説立案

集めた証拠から、**根本原因の仮説を2〜5個**並べる。各仮説について:

| 仮説 | 想定メカニズム | 確認方法 | 推定確度 |
|---|---|---|---|
| H1 | … が … のとき … になる | 〇〇を実行/読めば確かめられる | 高/中/低 |

仮説を**1つに絞らない**。最初の仮説に飛びつくと真因を見逃す。

---

## フェーズ4: 仮説の検証と「なぜ?」の連鎖

確度の高い順に仮説を検証する:

1. ログ追加 / デバッガ / `console.log` / `print` で**実際の値・実行順序**を観測
2. 仮説が外れたら **次の仮説に移る**（無理に押し通さない）
3. 当たったら、そこで止めず **「なぜそうなるのか?」を最低3回繰り返す**

例:
> - 症状: ユーザー削除後に再ログインできる
> - なぜ? → セッショントークンが失効していない
> - なぜ? → 削除処理がセッションテーブルを触っていない
> - なぜ? → user 削除と session 失効が別サービスにあり、削除イベントを購読していない
> - **根本原因**: イベント連携の設計漏れ（user.deleted を session-service が購読していない）

「コードの書き方の問題」ではなく **「設計・契約・データの問題」までたどり着いたら止まる**。

---

## 🛑 チェックポイント: 診断レビュー

根本原因が確定したら、**修正に着手する前に**ユーザーに以下を提示して承認を得る:

> ## 診断結果
> **症状**: <1行>
> **根本原因**: <1〜2文。なぜが3回以上続いた末の結論>
> **証拠**: <該当ファイル:行、ログ、コミットハッシュ>
> **修正方針**: <根本原因に対する直し方>
> **影響範囲（同じ原因の別ケース候補）**: <あれば列挙>
> **対症療法の選択肢があるか**: <ある場合は短期で済む案も書く。その場合の負債も明記>
>
> この診断で進めますか？

承認なしに修正に進まない。診断への異論があれば、追加の仮説検証に戻る。

---

## フェーズ5: 修正

承認後、根本原因に対する修正を行う:

- **根本原因のレイヤーで直す**（症状が出る末端で if 分岐しない）
- 修正は最小限の差分にする（無関係なリファクタを混ぜない）
- コミットメッセージ:
  - 件名: `fix: <症状を一言で>`
  - 本文: **症状 / 根本原因 / 修正内容** を分けて書く
  - 末尾: `Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>`

---

## フェーズ6: リグレッションテストの追加

このバグが**再発したら必ず CI で落ちる**テストを追加する:

- 既存のテストファイルに追加（命名規則を守る）
- テストは **修正前のコードで失敗し、修正後のコードで通る** ことを必ず確認
- E2E でしか再現できない場合は最低でも統合テスト相当を書く
- どうしてもテスト不能なら、その理由をコミットメッセージに明記

テスト追加なしで完了報告しない。例外は「コミットに修正不可能な物理 I/O が絡む」など、書けない明確な理由があるときのみ。

---

## フェーズ7: 兄弟バグの探索

同じ根本原因から **他にも症状が出ている可能性**を Explore で探す:

- 「同じイベント未購読 / 同じ契約違反が他にもないか」
- 「同じ関数を使っている別の呼び出し元の挙動」

見つかった候補は **自動修正しない**。一覧にして報告し、ユーザー判断を仰ぐ:

> ## 兄弟候補
> 同じ原因で壊れている可能性のある箇所:
> - `path/to/other.ts:42` — <短い説明>
> この場で一緒に直しますか？

---

## フェーズ8: 検証

最後に:

```bash
# フェーズ1の再現手順を再実行 → 症状が出ないことを確認
# 関連テスト一式を実行
```

合わせて、追加したリグレッションテストが通ることを確認する。

---

## 最終報告

```
## バグ修正完了: <症状の一言サマリ>

**根本原因**: <1〜2文>

**修正**:
- <commit hash> <subject>

**追加したリグレッションテスト**:
- `path/to/test.ts::test_name`

**兄弟候補（未対応）**:
- <あれば>

**確認**:
- 再現手順で症状が出ないこと: ✅
- 関連テストすべて通過: ✅
```

---

## 厳守事項

- **根本原因の特定なしに修正しない**。「動いたから OK」は失格。
- 「なぜ?」を最低3回繰り返してから根本原因と称する。
- 対症療法は明示してユーザー承認を得たときのみ採用。暗黙に握りつぶさない。
- 診断結果のユーザー承認なしに修正に進まない。
- リグレッションテストなしに完了報告しない。
- 兄弟バグは自動修正しない（報告のみ）。
