機能開発のエンドツーエンドフロー。**フェーズ1とフェーズ2の間に必ずユーザー承認のチェックポイントを置く**。

> 💡 **計画だけ欲しい場合**: フェーズ1完了時点で承認しなければ、計画が `.claude/plans/` に保存された状態で停止する。実装に進まないのが既定。

---

## フェーズ1: 要件ヒアリングと計画作成

### 1-1. ヒアリング

ユーザーから機能の概要を聞いたら、自分の言葉で**1〜2文で復唱**して認識を合わせる。

次に、以下のうち**まだ不明な点だけ**を質問する（既に明らかな項目は聞かない）。一度の質問は最大3つまで:

1. **インターフェース** — UI / API / CLI / バックグラウンドジョブ
2. **入出力** — 入力データの形式・出所、出力の形式・保存先
3. **エッジケース** — 想定する異常系、認可/権限、空入力、競合
4. **スコープ外** — 「今回はやらないこと」
5. **既存コードとの統合** — 似た機能、再利用すべきモジュール
6. **非機能要件** — 性能・スケール・観測性
7. **リリース粒度** — 1 PR か段階リリース、feature flag の要否

「いい感じに」「適切に」は禁句として扱い、具体化を求める。

### 1-2. コードベース調査（Explore サブエージェントに委譲）

要件が固まったら、**メインコンテキストを汚さないため**、調査は `Explore` サブエージェントに投げる。並列に複数投げてよい。

Explore に投げるべき調査の例（要件に応じて選ぶ）:

- 「`<関連キーワード>` を含むファイルと、類似機能の実装パターン（命名規則・ディレクトリ構成・テスト方法）」
- 「再利用可能な既存ユーティリティ・型・コンポーネント（`<ドメイン>` 領域）」
- 「`<変更対象モジュール>` の呼び出し元と依存先（影響範囲）」

ブランチ運用の流儀は軽量なので、メインで直接拾ってよい:

```bash
git branch -a 2>/dev/null | head -20
git log --merges -10 --oneline 2>/dev/null
```

Explore の返答と git の出力を統合して、「このリポジトリの流儀ではこう書く / こう切る」を1〜3行で要約する。**Explore の生出力を計画書にそのまま貼らない**（要約のみ反映）。

### 1-3. 計画書の保存

以下の構成で `.claude/plans/<kebab-case-slug>.md` に保存する。ディレクトリがなければ `mkdir -p` で作成。

~~~markdown
# <機能名>

## 目的
1〜2文。WHY を中心に書く（WHAT はスコープに分離）。

## スコープ
**やること:**
- ...

**やらないこと:**
- ...

## アプローチ
高レベルの設計判断。なぜこの方法を選んだか、他の選択肢をなぜ却下したか。

## ブランチ戦略

**方針: ブランチ/PR はできる限り細かく切る。** 1 PR は 1 つの論理単位（理想 ≤300行）。「機能 = 1つの大きな PR」にせず、**スキーマ / API / UI / 配線** のような責務で分割するのが既定。

- **ベースブランチ**: `main`（検出した既定）
- **ブランチ命名**: `<type>/<slug>`（例: `feat/user-auth-schema`、連番 `-01` `-02` で順序を示してもよい）
- **PR分割**: 原則複数 PR に分割。各 PR について以下を明記:
  - スコープ（その PR で何が完成するか）
  - 依存（どの PR がマージされてから出すか）
  - 単独でマージしても壊れないこと（半端な機能は feature flag や未配線で隠す）
- **単一 PR の例外**: 50行未満で完結 / 分割すると逆に意味が壊れる結合度
- **マージ戦略**: squash / merge commit / rebase（リポジトリの流儀に従う）
- **feature flag**: 段階リリース時はフラグ名と削除条件（いつ・誰が剥がすか）

## 変更ファイル
- `path/to/existing.ts` — 何を追加/変更するか（1行）
- `path/to/new.ts` (新規) — 役割（1行）

## 実装手順
1. ステップ1（依存のないものから順に）
2. ステップ2
3. ...

各ステップは独立してコミット可能な粒度。PR を分割する場合は `[PR1]` `[PR2]` でステップとPRの対応を明示する。

## テスト戦略
- ユニットテストで担保する範囲
- 統合テスト / E2E で担保する範囲
- 手動確認が必要な部分

## リスク・未確定事項
- リスクA → 緩和策
- 未確定事項B → 誰に / どう確認するか
~~~

保存後、**ファイルパスを報告**して次のチェックポイントに進む。

---

## 🛑 チェックポイント (必須)

- ユーザーから **「承認」「OK」「進めて」など明示の同意**を得るまで、絶対に次のフェーズに進まない。
- 「うーん」「ちょっと…」など曖昧な返答は **否承認** として扱い、何が引っかかったか聞き返す。
- 修正指示があれば計画ファイルを更新して、再度承認を求める。

承認メッセージ例:

> 計画を `.claude/plans/<slug>.md` に保存しました。
> 内容を確認して、進める場合は **「承認」** か **「OK」**、修正が必要なら指摘内容を返してください。

---

## フェーズ2: 実装

承認を受けたら、計画の「実装手順」に従って実装する:

- **ブランチ戦略に従ってブランチを切ってから着手**する（計画書の「ブランチ戦略」の通り）
- **1ステップ = 1コミット**の粒度で進める
- 各ステップ完了時に必要なら型チェック・テストを走らせる
- コミットメッセージ:
  - 命令形・件名 ≤72 文字・WHY 中心
  - 末尾に `Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>`
- **計画から逸脱が必要になったら止まってユーザーに確認**する（暗黙に方針変更しない）
- 全ステップ完了後、変更ファイル一覧とコミットハッシュを列挙する

---

## フェーズ3: セルフレビュー

実装完了後、`.claude/skills/review/SKILL.md` の観点で自分の変更をレビューする:

```bash
git diff main...HEAD
```

各指摘を以下に分類:

| 分類 | 含まれるもの | 自動修正 |
|---|---|---|
| **Critical** | バグ・セキュリティ脆弱性・データ整合性・既存機能のリグレッション・型エラー | **する** |
| **Warning** | 性能上の懸念・不十分なエラーハンドリング・可読性に問題のある箇所 | しない（報告のみ） |
| **Suggestion** | 命名・スタイル・小さなリファクタ案 | しない（報告のみ） |

---

## フェーズ4: Critical の自動修正

Critical が0件ならスキップ。1件以上あれば修正する:

- 1件ずつ Edit ツールで修正
- **1修正 = 1コミット**（`fix: <Criticalの要約>`）
- 全修正完了後、**もう一度だけ**フェーズ3を実行して Critical が0件になったことを確認
- 2巡目でも Critical が残る場合: **自動修正を打ち切り**、状況をユーザーに報告（無限ループ防止）

Warning / Suggestion は **自動修正しない**。最終報告にリストアップするだけ。

---

## フェーズ5: 計画ファイルのアーカイブ

最終報告の**直前に**、対応する計画ファイルを完了済みディレクトリへ移動する:

```bash
mkdir -p .claude/plans/completed
mv .claude/plans/<slug>.md .claude/plans/completed/$(date +%Y-%m-%d)-<slug>.md
```

- 日付プレフィックスで完了順を保持
- ユーザーが「実装は途中で中断」と明示した場合は **移動しない**（`.claude/plans/` 直下に残す）
- 同名ファイルが既にあれば連番を付ける（`-2.md`）

**計画書**: `.claude/plans/completed/YYYY-MM-DD-<slug>.md` にアーカイブ

**コミット**:

**残課題 (ユーザー判断)**:

### Suggestion
- ...

**手動確認推奨**:
- <自動テストでカバーできていない動作確認項目>

## 厳守事項

- **承認なしに実装に進まない**。これが唯一の絶対ルール。
- Critical の定義からはみ出すものを自動修正しない。
- 自動修正のループは最大2巡。
- 計画ファイルは実装中に**勝手に書き換えない**（逸脱が必要なら止まって確認 → 計画更新 → 再開）。
- フェーズ間で何をしているかを1行ずつ告げる（「フェーズ2: 実装を開始します」など）。
- フェーズ1〜1-3 ではコードを書かない。
