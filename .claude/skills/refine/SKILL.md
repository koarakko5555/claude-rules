このセッションの会話履歴を振り返り、`.claude/` 配下のドキュメント（と `CLAUDE.md` / `settings.json`）に反映すべき変更を抽出して提案する。**承認を受けた候補のみ適用する**。

---

## フェーズ1: 観測

会話履歴を最初から遡って、以下を抽出する。

### 拾うもの

1. **明示的な訂正** — 「やめて」「違う」「そうじゃなくて」を含む発言と、その直前の自分の行動
2. **非自明な判断の承認** — 「いいね」「採用」「そう」など、ユーザーが**驚かずに受け入れた選択**（特に分かれ道で選んだもの）
3. **繰り返しの強調** — 同じ要望が2回以上出たもの（例: 言語、粒度、フォーマット）
4. **質問で初めて表面化した前提** — ユーザーへの確認質問で引き出された制約・好み
5. **構造的な変更の根拠** — ファイル削除・リネーム・新規作成の背景にあった意図
6. **頻出 Bash コマンド** — このセッションで2回以上実行された Bash コマンド（特に許可プロンプトで毎回承認したもの）

### 拾わないもの

- 一度きりのタスク固有のやりとり
- 既にドキュメントに書かれている内容（必ず Read してから判断する）
- コードから自明に読み取れる事実
- 「念のため」「将来役立つかも」レベルの推測

---

## フェーズ2: 仕分け

抽出した候補を以下のいずれかに割り当てる:

| 候補の性質 | 行き先 |
|---|---|
| コーディングの好み・禁止事項 | `.claude/rules/coding.md` |
| Git / ブランチ / PR の運用 | `.claude/rules/git.md` |
| プロジェクト固有の事実・前提 | `./CLAUDE.md` |
| 新しい skill が欲しい | `.claude/skills/<name>/SKILL.md` (新規) |
| 既存 skill の手順変更 | `.claude/skills/<name>/SKILL.md` |
| 自動化フック・許可コマンド | `.claude/settings.json` の `permissions.allow` |
| 既存ファイルに合わない新カテゴリ | `.claude/rules/<新規>.md` (新規) |

仕分けが曖昧な候補は **保留** として別枠に出す。勝手にどこかへ押し込まない。

### settings.json の `permissions.allow` に追加する基準

毎回の承認プロンプトを減らすため、頻出コマンドを自動で許可リストに入れる候補にする。**ただし以下を厳守**:

**追加してよいもの（安全）**:
- 読み取り専用: `Bash(ls*)`, `Bash(cat*)`, `Bash(git status)`, `Bash(git diff*)`, `Bash(git log*)`, `Bash(find . *)`, `Bash(grep *)`
- 冪等な調査系: `Bash(npm test*)`, `Bash(pytest*)`, `Bash(tsc --noEmit*)`, `Bash(eslint*)`
- 既存 allow にあるパターンの**個別具体化**（例: 既に `git diff*` があるなら `git diff main` は不要）

**追加してはいけないもの（提案しない）**:
- 破壊的: `rm`, `mv`, `git reset`, `git push --force`, `DROP`, `TRUNCATE`
- 権限昇格: `sudo`, `chmod`, `chown`
- 外部送信: `curl POST`, `gh pr create/merge`, `git push`
- 引数次第で危険: `eval`, `xargs`, パイプの右側で削除を含むもの
- ワイルドカードが広すぎ: `Bash(*)`, `Bash(rm*)`, `Bash(npm*)` 全般

**提案の粒度**:
- できるだけ**具体的な接頭辞 + ワイルドカード**で（`Bash(npm run test*)` > `Bash(npm*)`）
- 既存の allow / deny エントリと**重複・矛盾しないか**必ずチェック

## フェーズ3: 差分提案

**書き換える前に**、各候補を以下の形式で1件ずつ提示する。最大5件まで。多すぎる場合は重要度で絞る。

~~~
### 候補 N: <一言サマリ>

**根拠**: <会話のどの部分から来たか。該当発言を短く引用>
**反映先**: `.claude/rules/coding.md`
**操作**: 追記 / 置換 / 新規作成 / 削除
**変更案**:
> （置換の場合は before → after、追記の場合は追記する内容、新規作成の場合は全文の要約）

**永続化する価値**: <一般化可能 / 繰り返し言及あり / 暗黙の前提 等のどれか>
~~~

提示後、**必ず停止**して以下のメッセージを出す:

> 以上 N 件の更新候補を提示しました。
> 適用する番号を **「1,3,4」** のように指定してください。
> 全部なら **「全部」**、なしなら **「なし」**、修正があれば指摘内容を返してください。

---

## フェーズ4: 適用

ユーザーから採用番号を受けたら、その候補のみを Edit / Write で適用する。

- 既存ファイルへの追記/置換は Edit を使う（既存内容を尊重）
- 新規ファイルは Write で作成
- `settings.json` を更新する場合:
  - JSON として valid なまま編集する（カンマ忘れ・余計なカンマに注意）
  - `permissions.allow` への追加は配列の末尾に追記する
  - 編集後、`node -e "JSON.parse(require('fs').readFileSync('.claude/settings.json','utf8'))"` か `python3 -c "import json; json.load(open('.claude/settings.json'))"` で構文チェック

適用後、以下を1メッセージで報告する:
- 変更したファイル一覧と各ファイルの追加/削除行数
- 適用しなかった候補と理由（ユーザーが「なし」と言った場合を除く）

---

## 厳守事項

- **ユーザー承認なしに `.claude/` 配下と `CLAUDE.md` / `settings.json` を書き換えない**。フェーズ3で必ず止まる。
- 候補は**最大5件**。「全部出す」ではなく重要度で絞る。
- 既存ルールとの **重複・矛盾** をチェックする（提案前に該当ファイルを Read する）。矛盾する場合は差し替え案として提示し、理由を添える。
- セッション固有の進行状況・todo は対象外。Plan / Todo の領域に踏み込まない。
- コード変更・実装は一切行わない（`.claude/` 配下のドキュメント更新のみ）。
